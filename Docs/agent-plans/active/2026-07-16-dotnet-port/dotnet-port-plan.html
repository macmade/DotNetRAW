Overview

SwiftRAW is a deliberately read-only Swift wrapper over LibRAW: it opens a RAW file, unpacks the sensor data, and surfaces the image geometry, colour calibration, lens identity, and shooting metadata plus the raw 16-bit Bayer buffer. It does not demosaic, decode thumbnails, or write. This plan ports that library to idiomatic C# targeting .NET 10, preserving the public API shape, type names, responsibilities, and observable behaviour, while adopting established .NET conventions.

The single largest difference from the original is the native binding. SwiftRAW reaches LibRAW through Swift/C++ interop plus a small C++ shim (SwiftRAWLibRaw.h) — a mechanism .NET has no equivalent for. As the source document directs, the port instead binds LibRAW's flat C API (libraw_init, libraw_open_file, libraw_open_buffer, libraw_unpack, libraw_recycle, libraw_close, libraw_strerror, libraw_version, libraw_get_iparams/lensinfo/imgother, …) via P/Invoke (DllImport + marshaling) against the bundled native library. This makes the native interop layer a new, high-risk, early milestone that has no counterpart in the sibling DotNetFITS port (which is pure-managed).

The approach mirrors DotNetFITS's proven shape: dependency-ordered, incremental milestones, each porting one layer with its tests (TDD), each leaving the solution compiling with all tests green so it can be reviewed and committed on its own. Conventions, formatting, the single-exception error model, culture-invariance rules, test infrastructure, and the "describe the library on its own terms" framing are all carried over from DotNetFITS.

Questions & answers

1. The bundled libraw.dll is Windows-only — how should native tests run across platforms?

Answer: ship a macOS .dylib. A macOS dynamic library will be produced from SwiftRAW's static LibRAW archive and committed alongside the Windows DLL, so the LibRAW-calling tests run on both the development Mac and Windows CI. The maintainer will update SwiftRAW's macOS LibRAW build to 0.22.2 so the macOS .dylib matches the bundled Windows DLL version exactly — keeping the version-string/number tests uniform across platforms rather than OS-dependent.

2. Swift's RAWFile frees its native context deterministically in deinit — how should the port manage native lifetime?

Answer: IDisposable + SafeHandle. A LibRawHandle : SafeHandle owns the native libraw_data_t* and calls libraw_recycle + libraw_close in ReleaseHandle. RAWFile implements IDisposable (deterministic release with a finalizer safety net) and, for the Data/byte-buffer path, pins and owns the input buffer for the file's whole lifetime (LibRAW's open_buffer retains the caller's pointer). As an idiomatic C# addition Swift lacks, RAWFile throws ObjectDisposedException once disposed.

3. How should the ~14 immutable metadata value types be represented?

Answer: sealed record classes. These are per-file, low-cardinality aggregates (materialised a handful of times per opened file, not in bulk), and several already hold heap-allocated arrays/strings — so the readonly struct allocation argument that justified FITSValue in DotNetFITS does not apply here (a struct would remove at most one small header object while the real allocations remain, and several of these types are large enough that value-copy costs outweigh it). sealed record gives immutability, value equality, and a clean ToString() override. Because record equality compares array fields by reference, the two array/matrix-bearing types (RAWColorData, RAWCFAPattern) get custom structural Equals/GetHashCode to preserve Swift's deep Equatable. RAWParsingOptions (a single Bool in Swift) is a natural readonly record struct; the nested Kind/Layout/FocalType are plain enums.

Conventions (carried over from DotNetFITS)

  • Namespaces are flat — namespace DotNetRAW; and namespace DotNetRAWTests; (file-scoped). Subfolders are organisational only and do not become namespace segments, so the public API stays recognisable to SwiftRAW users.
  • One public type per file, grouped into subfolders that mirror SwiftRAW's own layout: Interop/ (new — the native layer), Core/, Image/, Color/, Lens/, Metadata/, Extensions/. Test folders mirror the library folders.
  • .NET naming, SwiftRAW formatting. PascalCase members, I-prefixed interfaces — but keep SwiftRAW's house formatting: Allman braces, spaces inside parentheses/brackets/generics and attributes ([ Fact ], new byte[ 32 ], IReadOnlyList< int >, $"… { value } …"), aligned = across consecutive assignments and tabular-aligned columns, empty bodies as {}. These are applied by hand (not analyzer-gated).
  • XML doc comments on every type and member — including internal and private ones. GenerateDocumentationFile=true gates public members via CS1591; TreatWarningsAsErrors=true.
  • Describe the library on its own terms. Code, doc-comments, and README must not reference the Swift original, Foundation, or Swift idioms (a LibRAW dependency note in the README is fine — LibRAW is a real runtime dependency here, unlike FITS).
  • Prefer get/set (init-only) properties over plain fields, even private ones — record positional/explicit properties satisfy this.
  • MIT license header block atop every source file (the maintainer's standard block).

Interop strategy & the layout risk

The port binds LibRAW's flat C API. SwiftRAW reads eleven nested LibRAW structs. Three of them are reachable through dedicated C getters that return interior pointers — libraw_get_iparams (idata), libraw_get_lensinfo (lens), libraw_get_imgother (other) — so no offset maths is needed for those. The remainder (sizes, color, shootinginfo, makernotes.common, rawdata) have no getters and must be read from the libraw_data_t base pointer at their correct offsets. Most of these sit after the very large makernotes/params/rawparams sub-structs.

#1 risk — byte-exact struct layout. libraw_data_t is >150 KB and contains huge inline arrays (ushort curve[65536], two unsigned cblack[4104]). To reach color/shootinginfo/makernotes.common/rawdata the preceding sub-structs must be sized exactly. Recommended approach: model the sub-structs SwiftRAW actually reads field-by-field, and model every intervening sub-struct as a correctly-sized opaque byte block ([MarshalAs(UnmanagedType.ByValArray, SizeConst=N)]), then Marshal.PtrToStructure the whole libraw_data_t once per file. Validate all offsets/sizes with Marshal.SizeOf/Marshal.OffsetOf against expected values (cross-check on Windows, where the DLL matches the vendored headers). Use natural packing — do not force Pack=1. This is the single most error-prone milestone (M2).

Marshaling & porting hazards

  • No C++ interop / no shim. The C++ LibRaw methods SwiftRAW calls map 1:1 to the flat C functions (open_file→libraw_open_file, etc.). The shim's cblack accessor becomes a direct read of the first four cblack[] values from the marshaled color.
  • Fixed C char[N] buffers (make/model/software/serials/desc/…) → [MarshalAs(ByValArray, SizeConst=N)] byte[], decoded UTF-8 up to the first NUL (the port's equivalent of Swift's String(cCharArray:)). Distinguish these from the few genuine pointers (xmpdata, thumb, profile).
  • 2-D C arrays (rgb_cam[3][4], cam_xyz[4][3], cmatrix[3][4], xtrans[6][6], GPS latitude[3], …) are flat row-major buffers — reshape to T[][] via the ported CFixedArray helper.
  • ushort/unsigned signedness: sensor dimensions, pitch, black levels, filters, and the Bayer buffer are unsigned — use ushort/uint.
  • time_t timestamp is 64-bit on both the Windows x64 and macOS builds — marshal as long; 0 → null timestamp.
  • Raw buffer length = raw_height × (raw_pitch / 2) samples; return null when raw_image is null or pitch/height ≤ 0 (mirrors SwiftRAW).
  • Culture-invariance. SwiftRAW does no explicit locale work but relies on Swift's locale-independent interpolation. C# ToString() is culture-sensitive, so every numeric rendering in a description must use CultureInfo.InvariantCulture. Add a repo-root .editorconfig raising the globalization analyzers (CA1305, CA1304, CA1310, CA1309, …) to errors, and a culture-swap test (run under fr-FR) wherever a float/double is formatted. Preserve the exact Unicode characters in descriptions: × (U+00D7), – (en dash), — (em dash).
  • Reciprocal shutter formatting: denominator = round(1/shutter), guarded by isFinite && < 1e9, rendered "1/{n}s"; tolerate Infinity/NaN exposure values.
  • No deterministic deinit → IDisposable + finalizer + SafeHandle (see Q&A #2).
  • init? failable (RAWGPSInfo when gpsparsed==0) → a nullable static factory returning RAWGPSInfo?.

Native library packaging

  • Commit the natives under a per-RID layout — runtimes/win-x64/native/libraw.dll (per-architecture, Windows) and runtimes/osx/native/libraw.dylib (a single universal arm64 + x86_64 dylib under the generic osx RID, covering both Apple Silicon and Intel Macs) — and ensure the correct one is copied next to the test assembly and loaded at runtime. A NativeLibrary.SetDllImportResolver mapping the import name "libraw" to the per-RID file is the robust option if default probing is insufficient.
  • DllImport uses library name "libraw" with CallingConvention.Cdecl; ANSI/UTF-8 for the const char* file-path and error-string entry points. ([LibraryImport] source-generated marshaling is the modern .NET 10 alternative and may be used where the signatures are blittable; the source document names DllImport.)
  • The class library likely needs <AllowUnsafeBlocks>true</AllowUnsafeBlocks> for pointer/span work; the existing InternalsVisibleTo, GenerateDocumentationFile, CheckForOverflowUnderflow, and TreatWarningsAsErrors settings are kept.

LibRAW version

The bundled binaries are LibRAW 0.22.2 (numeric 0x1602 = 5634). SwiftRAW's tests assert "0.22.1-Release"; the port's version tests must therefore expect "0.22.2-Release" and 5634. Both platforms will report 0.22.2 (the macOS .dylib is being rebuilt to match), so the assertion is platform-uniform.

Implementation notes

The highest-risk layer, built TDD-style: the layout-validation tests were written to lock every offset, then the structs implemented to satisfy them. Green in both Debug and Release (14 tests, 0 warnings).

  • Ground truth from the headers. Every struct size and field offset was computed from the vendored LibRAW 0.22.2 headers with a clang offsetof/sizeof probe, compiled for both arm64 and x86_64 and diffed — byte-identical. Because none of the modelled regions contain a C long (whose width differs between Windows LLP64 and Unix LP64) and time_t/pointers/UINT64 are 8 bytes everywhere, the layout is architecture-invariant, so the arm64 numbers are valid for win-x64 too. Key values: libraw_data_t is 381576 bytes; within it sizes@8, idata@192, lens@632, makernotes.common@4784, shootinginfo@5088, color@5592, other@192680, rawdata@193768.
  • Bindings. Interop/LibRaw.cs now binds the full flat-C API via source-generated [LibraryImport] with [UnmanagedCallConv] cdecl. The context handle and every const char* return are IntPtr — never marshaled as string — so the marshaller never frees LibRAW-owned memory. open_file uses StringMarshalling.Utf8; open_buffer uses nuint for size_t.
  • SafeHandle. LibRawHandle releases via libraw_recycle then libraw_close (mirroring SwiftRAW's deinit), with IsInvalid on a null pointer. Per review, libraw_init is bound to return a LibRawHandle directly, so the source-generated marshaller creates and initialises the handle atomically (no allocate-then-SetHandle leak window).
  • Errors. LibRawError enum with the exact codes from libraw_const.h, plus IsFatal (< -100000).
  • Structs. Twelve flat marshalable structs for the sub-structures the port reads. Each is read from the context base pointer at its offset (constants in LibRawData) rather than marshalling the whole 372 KB context. color is modelled from its black field, skipping the leading 128 KB curve and 16 KB cblack; the four cblack values the port surfaces are read separately. GPS is read through the ParsedGps nested in other.
  • Uniform sequential layout. Every struct uses LayoutKind.Sequential, so the marshaller computes the offsets and a mistyped or misordered field shifts them and fails the guard. Unused intervening fields are modelled only to preserve the layout. (An initial Explicit [FieldOffset] attempt for the sparse structs was abandoned: the CLR rejects object-reference fields at non-8-aligned offsets under explicit layout, and — per review — explicit offset tests are tautological. Sequential resolves both.)
  • Signedness preserved exactly. filters unsigned (the CFA decode uses an unsigned shift), colors/flip signed, FocalType signed short, GPS refs and xtrans signed sbyte, black/maximum/cblack unsigned. The two same-named FocalLengthIn35mmFormat (a ushort in the EXIF lens, a float in the maker-note lens) are kept distinct.
  • Validation. The layout tests assert every struct's field offsets and sizes against the probe values, and additionally cross-check — against the actually loaded native library — that the three getter interior-pointers (libraw_get_iparams/lensinfo/imgother) land at exactly the declared offsets. A round-trip smoke test covers init→non-null→close, versionNumber() == 5634, strerror decoding, and the error codes/IsFatal.
  • Deviation from the plan's suggested approach. Rather than modelling the whole libraw_data_t with opaque ByValArray blocks and marshalling it once, the port reads each needed sub-struct at its offset — avoiding a 372 KB copy and the opaque-block alignment hazard. The getters the plan lists are bound and used for the runtime offset cross-check. CS0649 is suppressed per interop file (fields are populated by the marshaller).

Dependency order

native packaging (M1) → interop bindings + structs + SafeHandle (M2) → helpers + test infra (M3)
  → errors, options, version (M4)
    → image sizes/info (M5) → CFA + sensor (M6) → colour (M7) → lens (M8) → shot/GPS/camera (M9)
      → RAWFile (M10) → integration & parity (M11) → finalization (M12) → nuget package (M13)

M5–M9 are independent of one another (each ports one concern's records) and can be reordered, but all depend on M2 (marshalable structs) and M3 (helpers). M10 depends on M2–M9. Each milestone leaves the solution compiling with all tests green so it can be reviewed and committed alone.

LibRAW C entry points used by the port

Lifecycle/open/unpack: libraw_init, libraw_open_file, libraw_open_buffer, libraw_unpack, libraw_recycle, libraw_close. Getters: libraw_get_iparams, libraw_get_lensinfo, libraw_get_imgother. Diagnostics/version: libraw_strerror, libraw_version, libraw_versionNumber. Direct reads off the marshaled libraw_data_t: sizes, idata, lens, other, other.parsed_gps, shootinginfo, color, makernotes.common, rawdata (incl. rawdata.raw_image).
